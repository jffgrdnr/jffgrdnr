UX Guidelines

Forms handle all user inputs and form validation. We encourage developers to use the standardized input fields, select dropdowns, textareas, and validation.

Proper Usage

Guidelines for web form design.

  • Labels should be placed above the form fields.
  • Labels can be placed inside text fields in special circumstances. (For example: person lookups on the Tree Viewer) However, always place field labels outside text boxes on multiple field forms. When using labels inside fields, the field should always reset to the label if the user blurs the field and no data is present.
  • Form fields should employ default values when:
    • The user has previously entered a value (such as in form validation or editing set values)
    • The number of choices on the form is overwhelming to the point of overcomplexity
    • You have a strong qualified guess as to what most users will choose/want
    • The user has already chosen enough to calibrate the default option based on another choice they’ve entered.
  • Dropdowns should either display the default option or should display the text "Choose..."
  • There should only be one primary action button per form. If you have multiple action choices, use a text link or a brown button for the secondary action.
  • Checkboxes and radio buttons can be either vertically stacked or side-by-side. If there are just a few, short options (e.g. Male, Female, Unknown) use side-by-side layout. For everything else, use vertically stacked layout for options.

Required & Optional Fields

  • Required fields are denoted using a red asterick

Default Style Rules

  • Labels are bold #333 13px (Exception: labels next to radio/check inputs are not bolded)
  • Fields are regular 13px #333

Coding Guidelines

Prerequisites

  • CSS: /css/v1/main.css, /css/v1/button.css, /css/v1/form.css, /css/v1/grid.css, /css/v1/icon.css
  • JS: /js/v1/form.js
  • The [lang] attribute must be on the <html> element with its correct language and country code. (It technically can be on the <form> element as well, but this is not recommended).
  • Optional prerequisites based on usage:
    • CSS: /css/v2/alert.css, /css/v1/callout.css
    • JS: /js/v2/alert.js, /js/v1/callout.js

Instructions

  1. Forms should be laid out using our responsive grid.
  2. The <form> element should have the class form to trigger the newer styles. This exists to avoid conflicts with older pages.
  3. Labels should be actual <label> elements that are tied to their respective fields using the [for] attribue.
  4. <input />, <select>, and <textarea> elements should all have unique IDs for screen readers to be able to announce the correct <label> to the correct field.

Example #1 - Basic Form

  1. Form fields go 100% of width when using the .form class. This makes it extremely easy to use the grid for form layout.
  2. Take note that the grid allows you to have multiple "rows" of fields inside of a single parent container by taking advantage of the class .ancColrow.
  3. Also take note that the grid allows you to jump not only to 100% of the width of your container, but also to 33% and 50% by using the classes half768, half480, half320, third768, third480, third320. When using these classes, you'll need to mark the field that beings a new row at a certain breakpoint by using the one of these corresponding classes: .ancColRow768, .ancColRow480, .ancColRow320. See the grid for more examples.
  4. Remember when using the classes to change the widths of the columns that you should either use a single "full" class on the parent container of the columns or use the "full/half/third" classes on each individual column.
  5. Add the class textareaLarge for larger height textareas.

Example #2 - Different Width @ Different Breakpoints

  1. There are 3 supported form sizes: the default size, .formLarge, and .formLargest. These are in addition to the form class.
  2. On most forms it is recommended to bump up the field size from the default size to the large size. This increases the hit area for mobile devices making it easier to focus on a field. This is done by adding the class .formLarge480 in addition to the .form class.
  3. The .formLarge class should only be used when there are 3-6 fields.
  4. The .formLargest class should only be used when there are 1-3 fields.

Instructions - Validation

  1. Form validation is done with our custom unobtrusive jQuery plugin.
  2. Instantiating the $().validator() plugin also adds multiple polyfills that add cross-browser support for: placeholder text, URL inputs, and styled file inputs.
  3. There are many ways to augment the validator with certain CSS classes and some custom [data] attributes, but these are not required.

Example #4 - Simplest possible validation

  1. As shown above, you instantiate our "validator" plugin on the form itself, like so: $('#myFormId').validator();

  2. For this plugin, you must pass in a few options. There are 4 available options:

    fields

    Object, a list of the all the fields in the form to validate with the options for each field. There are a lot of available options for each field:

    required
    Boolean or Integer, whether field is required or number of checkboxes that must be selected
    errorLocation
    String, where to put the error message. - null, 'below', 'callout'
    pattern
    String or Function - 'number', 'url', 'tel', 'email', function(value, $field, required) { ... return true || false; }
    Important: When using a custom function, the required parameter must be handled by the function itself. The required parameter is passed in as the 3rd parameter to custom functions.
    maxLength
    Integer, max number of characters allowed in field
    minLength
    Integer, min number of characters allowed in field
    extensions
    Array of Strings, allowed file extentions for input[type="file"]. Must be lowercase. (i.e. ['png', 'jpg', 'gif'])
    when
    String, when to validate - any event. Default is on blur.
    collection
    jQuery Object, used for radio/checkbox inputs. Select each input in the group. i.e. $('input[name="interests[]"]')
    match
    String, pass in ID of the field to match. i.e. password field
    validateOnSubmitHover
    Boolean, default true
    submitButton
    jQuery Object, default $form.find('input[type="submit"]')
    onSubmit(valid, $form)
    Function, default false. This function is called after the entire form has been validated (and marked as such) and is ready to be submitted. The valid parameter tells you if the entire form is valid up to this point. Return true to submit the form and return false to prevent the form from submitting. This allows you to do custom validation of multiple fields prior to submission.
    allowMultipleSubmissions
    Boolean, default false - Forms generally are only ever submitted a singular time. If your form is using ajax to handle the submission, and you want the user to be able to submit the same form multiple times, set this option to true.

Example #5 - Advanced validation

Externally exposed validator methods

$.validator.match(value, match)
$.validator.maxLength(value, maxLength)
$.validator.minLength(value, minLengthh)
$.validator.extensions(value, extensions)
$.validator.email(value)
$.validator.number(value)
$.validator.tel(value)
$.validator.url(value)
These each return a boolean of whether the pattern is met.

Each field has these custom functions

$('#yourField').trigger('validation');
Triggers the previously bound validation on that field. This can be useful when you want some non-standard event to trigger validation on a field.
$('#yourField').trigger('validation-message', [isValid, indexOfErrorMessage]);
Ability to allow for a delayed callback to display the correct error message.
isValid: true, false, 'clear'
indexOfErrorMessage: Optional integer to set the error index of the message to display
$('#yourField').trigger('validation-error-index', [indexOfErrorMessage]);
Ability to easily set the error index of the error message to display
indexOfErrorMessage: Integer to set the error index of the message to display
$('#yourField').trigger('validation-remove');
Completely remove all validation from a field including error and success messages

Sliders

Sliders are essentially a more mobile-friendly version of a checkbox.

Prerequisites

  • CSS: /css/v1/slider.css
  • JS: /js/v1/slider.js
  • Optional: /js/v1/form.js (for validation)

Instructions

Guidelines for using sliders.

  • Sliders are to be used like a checkbox, in cases where you need to the user to select a "true / false", "accept / decline" or "yes / no" option.
  • Sliders may be used inside or outside a <form>.
  • The on / off states of the slider must be set with the data attributes: data-label-on="[name]" and data-label-off="[name]"
  • Sliders are auto-instantiated by slider.js, unless:
    • A data attribute of data-auto-instaniate="off" is set
